5 # Raw data tools
6 
7 ## HBFUtil
8 
9 Utility class for interaction record -> HBF conversion and sampling of IRs for which the HBF RDH should be added to the raw data from the CRU.
10 Should be used for MC->raw conversion to ensure that:
11 
12 * all detectors set the HBF and TF flags in a synchronized way
13 
14 * all TFs are complete (contain nominal /D=256/ number of HBFs with contiguously incremeneting orbit numbers):
15 for any given `payloadIR` (InteractionRecord = orbit/bc) for which payload is being created,
16 ```cpp
17 using IR = o2::InteractionRecord;
18 HBFUtil::fillHBIRvector(std::vector<IR>& dst, const IR& fromIR, const IR& payLoadIR);
19 ```
20 will fill the provided `dst` vector with the IR's (between `fromIR` and `payLoadIR`) for which the HBFs should be
21 generated and flushed to raw data **before** filling the payload of `payLoadIR`.
22 
23 See `Detectors/Raw/test/testHBFUtils.cxx` for details of usage of this class (also check the jira ticket about the [MC->Raw conversion](https://alice.its.cern.ch/jira/browse/O2-972)).

25 ## RawFileReader
26 
27 A class for parsing raw data file(s) with "variable-size" CRU format.
28 For very encountered link the DPL `SubSpecification` is assigned according to the formula used by DataDistribution:
29 ```cpp
30 SubSpec = (RDH.cruID<<16) | ((RDH.linkID + 1)<<(RDH.endPointID == 1 ? 8 : 0));
31 ```
32 This `SubSpecification` is used to define the DPL InputSpecs (and match the OutputSpecs).
33 A `FATAL` will be produced if 2 different link (i.e. with different RDH.feeId) will be found to have the same SubSpec (i.e. the same cruID, linkID and PCIe EndPoint).
34 Also, a `FATAL` is produced if the block expected to be a RDH fails to be recognized as RDH.

39 ## Raw data file checker (standalone executable)
40 
41 ```cpp
42 Usage: o2-raw-filecheck [options] file0 [... fileN]
43 Options:
44  -h [ --help ] print this help message.
45  -v [ --verbosity ] arg (=0) 1: print RDH on error, 2: print all RDH
46  -s [ --spsize ] arg (=1048576) nominal super-page size in bytes
47  -t [ --hbfpertf ] arg (=256) nominal number of HBFs per TF
48 ```
49 
50 Allows to check the correctness of CRU data (real or simulated) stored in the binary file.
51 Multiple files can be checked, with each file containing data for the same or distinct group of links.
52 
53 Apart from the eventual `FATAL` produced for unrecognizable RDH or 2 links with the same cruID, linkID and PCIe EndPoint but different feeId (see `RawFileReader`),
54 the following errors are reported (as `ERROR`) for every GBT link while scanning each file (the error counter of each link is incremented for any of this errors):
55 
56 * RawDataHeader packet counter (`RDH.packetCounter`) is not incremented by 1
57 * RawDataHeader page counter (`RDH.pageCnt`) is not incremented by 1
58 * New HBF starts with `RDH.pageCnt > 0`
59 * `RDH.stop` is set at the very 1st page of the HBF
60 * New HBF starts while the previous one was not yet closed (no page `RDH.stop = 1` received)
61 * Number of HBFs in TF differs from the nominal (D=256, can be imposed by user)
62 * The orbit/BC of the new HBF does not differ from previous one by 1 orbit exactly
63 * New TF starts not from the beginning of the CRU SuperPage (not reported for single-link files since in that case there is no way delimit a superpage)
64 
65 After scanning each file, for every link the largest SuperPage size (as the size of the largest contiguos block containing data of this link).
66 A warning is printed if this size exceeds the nominal SuperPage size (D=1MB, can be imposed by user).
